# Finance CRM

**Finance CRM** - це внутрішня CRM-система для страхової компанії, розроблена як pet project / дипломний проєкт.  
Система орієнтована не на клієнтський кабінет, а на **внутрішню роботу співробітників**: менеджерів, керівника відділу та адміністратора.

Основний інтерфейс реалізований у вигляді адміністративної панелі на **Filament**.

---

## Призначення проєкту

Проєкт моделює внутрішню роботу страхової компанії та покриває базові CRM-процеси:

- ведення клієнтської бази;
- оформлення та супровід страхових полісів;
- облік оплат за полісами;
- реєстрація страхових випадків;
- обробка вхідних заявок;
- розмежування доступу за ролями;
- базовий контроль активності та операцій у системі.

Система не містить повноцінної клієнтської публічної частини, але вже має підготовлену внутрішню модель для майбутнього лендингу з формою заявки.

---

## Основна ідея системи

У системі передбачено три основні ролі:

- **admin** - повний доступ до системи;
- **supervisor** - керівник відділу;
- **manager** - менеджер.

Логіка доступу побудована навколо ролей:

- менеджер працює лише зі **своїми клієнтами**;
- менеджер бачить лише **свої поліси**, оплати, вхідні заявки та пов’язані страхові випадки;
- supervisor бачить загальну картину по всіх менеджерах;
- admin має повний контроль над системою;
- страхові довідники доступні для перегляду всім ролям, але **редагування, створення та видалення** доступні лише **admin** та **supervisor**.

---

## Основні модулі

### 1. Клієнти

Система підтримує два типи клієнтів:

- **фізична особа**
- **компанія**

Для клієнтів зберігаються:

- тип клієнта;
- статус;
- ПІБ або назва компанії;
- контактні дані;
- номер документа;
- ІПН / ЄДРПОУ;
- дата народження (для фізичної особи);
- джерело звернення;
- закріплений менеджер;
- нотатки.

### 2. Вхідні заявки

Окремий модуль для роботи з новими зверненнями:

- фіксація заявки;
- призначення менеджера;
- зміна статусу заявки;
- конвертація заявки в клієнта.

Цей модуль можна використати як основу для майбутньої публічної форми заявки з лендингу.

### 3. Поліси

Для кожного полісу зберігаються:

- номер полісу;
- клієнт;
- страхова пропозиція;
- менеджер;
- дата початку дії;
- дата завершення дії;
- сума оплати;
- сума покриття;
- строк оплати;
- статус полісу;
- комісія;
- службові нотатки.

### 4. Оплати полісів

Для оплат ведеться облік:

- методу оплати;
- статусу оплати;
- строку оплати;
- суми;
- транзакції;
- службових дат оплати / ініціації.

Статус полісу обчислюється централізовано на основі правил бізнес-логіки та стану оплат.

### 5. Страхові випадки

Модуль страхових випадків дозволяє:

- створювати заяву по активному полісу;
- фіксувати дату події;
- вказувати місце, причину та опис;
- вносити заявлену, резервну та виплачену суму;
- вести нотатки по заяві;
- відстежувати статус кейсу.

### 6. Журнал активності

У системі реалізований журнал активності, який дозволяє:

- логувати створення, оновлення та видалення записів;
- бачити історію змін;
- обмежувати видимість логів відповідно до ролі користувача.

### 7. Страхові довідники

Окремий блок довідників:

- категорії страхувань;
- страхові компанії;
- страхові продукти;
- страхові пропозиції.

Для **manager** довідники доступні лише в режимі перегляду.  
Для **admin** та **supervisor** доступні створення, редагування та видалення.

## Дашборд

У системі реалізовано рольовий дашборд.

### Для manager:
- кількість власних клієнтів;
- кількість власних полісів;
- кількість власних страхових випадків;
- прострочені оплати;
- динаміка клієнтів і полісів;
- полісів, строк дії яких скоро завершується.

### Для supervisor / admin:
- загальні показники по системі;
- динаміка клієнтів і полісів;
- розподіл полісів за статусами;
- аналітика по менеджерах;
- таблиця полісів, строк дії яких скоро завершується.

## Бізнес-логіка статусів

### Статус полісу

Статус полісу визначається автоматично за правилами:

- якщо є оплачений платіж і строк дії полісу ще не завершено -> **active**
- якщо є оплачений платіж і строк дії полісу завершено -> **completed**
- якщо оплати немає і строк оплати минув -> **canceled**
- в інших випадках -> **draft**

### Статус оплати

Оплата залежить від методу та стану проведення:

- `draft`
- `scheduled`
- `paid`
- `overdue`
- `canceled`

## Приклад бізнес-процесу

1. Менеджер отримує вхідну заявку або створює клієнта вручну.
2. Заявка призначається відповідальному менеджеру.
3. На основі заявки формується картка клієнта.
4. Менеджер обирає страхову пропозицію та створює поліс.
5. Система автоматично встановлює строк оплати.
6. Менеджер або відповідальний співробітник фіксує оплату.
7. Система автоматично оновлює статус полісу.
8. У разі страхового випадку створюється заява та ведуться нотатки по кейсу.
9. Supervisor або admin контролює загальну картину в системі.

## Встановлення та запуск

### 1. Клонування репозиторію

```bash
git clone https://github.com/meowam/finance-crm.git
cd finance-crm
```

### 2. Встановлення PHP-залежностей

```bash
composer install
```

### 3. Встановлення frontend-залежностей

```bash
npm install
```

### 4. Підготовка середовища

```bash
cp .env.example .env
php artisan key:generate
```

### 5. Налаштування бази даних

Для **тестів** у проєкті вже використовується **SQLite in-memory** через `phpunit.xml`, тому окремо нічого налаштовувати не потрібно.

Для **локального запуску з демо-даними / seeders** рекомендовано використовувати **MySQL**.

У `.env` потрібно вказати:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=finance
DB_USERNAME=root
DB_PASSWORD=
```

Після цього виконати міграції:

```bash
php artisan migrate
```

### 6. Наповнення тестовими даними

```bash
php artisan db:seed
```

### 7. Запуск проєкту

Зручно запускати так:

```bash
composer run dev
```

Або окремо:

```bash
php artisan serve
npm run dev
```
