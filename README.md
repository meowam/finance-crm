# Finance CRM

**Finance CRM** - це навчальний проєкт, який моделює роботу страхової компанії.  
Проєкт складається з двох основних частин:

- публічного лендингу страхової компанії з формою заявки;
- внутрішньої CRM-панелі для співробітників компанії.

Система орієнтована на демонстрацію повного базового процесу: від заявки з сайту до створення клієнта, оформлення полісу, обліку оплати та супроводу страхового випадку.

Основний внутрішній інтерфейс реалізований у вигляді адміністративної панелі на **Filament**.

---

## Призначення проєкту

Проєкт моделює внутрішню роботу страхової компанії та покриває базові CRM-процеси:

- прийом заявок з публічного лендингу;
- автоматичне призначення заявки менеджеру;
- ведення клієнтської бази;
- конвертація вхідної заявки в клієнта;
- оформлення та супровід страхових полісів;
- облік оплат за полісами;
- автоматичний перерахунок статусів оплат і полісів;
- реєстрація страхових випадків;
- ведення нотаток по страхових випадках;
- розмежування доступу за ролями;
- базовий контроль дій користувачів у системі;
- перегляд управлінської аналітики на дашбордах.

Проєкт не є реальною комерційною системою і не містить інтеграції з платіжними провайдерами, SMS-сервісами або зовнішніми страховими API.  
Платіжна логіка та частина бізнес-процесів реалізовані як демо-механізми для демонстрації роботи системи.

---

## Основна ідея системи

У системі передбачено три основні ролі:

- **admin** - технічне адміністрування системи: керування користувачами, довідниками, службовими налаштуваннями та проблемними записами;
- **supervisor** - керівник відділу, який контролює роботу менеджерів і переглядає управлінську аналітику;
- **manager** - менеджер, який працює з власними клієнтами, заявками, полісами, оплатами та страховими випадками.

Логіка доступу побудована навколо ролей:

- менеджер працює лише зі своїми клієнтами;
- менеджер бачить лише свої поліси, оплати, вхідні заявки та пов’язані страхові випадки;
- менеджер може працювати з нотатками лише по страхових випадках, пов’язаних із його полісами;
- supervisor бачить загальну картину по всіх менеджерах і має доступ до управлінської аналітики;
- admin відповідає за технічне адміністрування системи, користувачів, довідники та контроль проблемних записів;
- страхові довідники доступні для перегляду всім ролям, але створення, редагування та видалення доступні лише admin та supervisor.

## Публічний лендинг

У проєкті реалізовано публічний лендинг страхової компанії.

Лендинг доступний за адресою:

```
/
```

На лендингу є:

- опис страхових послуг;
- блок переваг;
- блок процесу роботи;
- адаптивне меню;
- форма заявки на консультацію;
- перемикання типу заявника: фізична особа / компанія;
- frontend-логіка для форми та мобільної навігації.

Форма заявки надсилається на маршрут:

```
POST /lead-request
```

Після відправки форми система:

- валідує дані заявника;
- нормалізує номер телефону;
- створює запис у модулі вхідних заявок;
- встановлює джерело заявки `landing`;
- встановлює статус заявки `new`;
- автоматично призначає заявку активному менеджеру з найменшим навантаженням;
- надсилає менеджеру внутрішнє сповіщення про нову заявку.

Для захисту від надмірної кількості запитів на форму використовується throttle-обмеження.

## Основні модулі

### 1. Клієнти

Система підтримує два типи клієнтів:

- фізична особа;
- компанія.

Для клієнтів зберігаються:

- тип клієнта;
- статус;
- ПІБ або назва компанії;
- контактні дані;
- номер документа;
- ІПН / ЄДРПОУ;
- дата народження для фізичної особи;
- джерело звернення;
- закріплений менеджер;
- нотатки.

Статус нового клієнта за замовчуванням встановлюється як `lead`.

Якщо для клієнта зі статусом `lead` створено оплачений поліс і поліс переходить у статус `active`, клієнт автоматично переводиться у статус `active`.

Джерела звернення клієнтів у системі уніфіковані:

- `landing` - заявка з лендингу;
- `manual` - створено вручну;
- `recommendation` - рекомендація.

### 2. Вхідні заявки

Модуль вхідних заявок використовується для обробки нових звернень.

Заявки можуть бути:

- створені через публічний лендинг;
- створені вручну у CRM;
- отримані як рекомендація.

Для заявки зберігаються:

- тип заявника;
- ПІБ або назва компанії;
- телефон;
- email;
- напрям інтересу;
- джерело заявки;
- статус;
- коментар;
- відповідальний менеджер;
- прив’язаний клієнт після конвертації.

Основні статуси заявки:

- `new` - нова;
- `in_progress` - опрацьовується;
- `converted` - конвертовано у клієнта;
- `rejected` - відхилено.

Заявку можна конвертувати у клієнта.  
Після конвертації повторне створення клієнта з тієї самої заявки блокується.

### 3. Поліси

Для кожного полісу зберігаються:

- номер полісу;
- клієнт;
- страхова пропозиція;
- менеджер;
- дата початку дії;
- дата завершення дії;
- сума оплати;
- сума покриття;
- строк оплати;
- статус полісу;
- комісія;
- службові нотатки.

Після створення полісу система автоматично створює початковий draft-платіж без вибраного методу оплати.  
Це дозволяє відображати поліс у внутрішньому процесі ще до фактичного вибору способу оплати.

Статус полісу не задається вручну як довільне значення, а перераховується на основі строків дії полісу та стану оплат.

### 4. Оплати полісів

Для оплат ведеться облік:

- полісу;
- методу оплати;
- статусу оплати;
- строку оплати;
- суми;
- номера транзакції;
- дати ініціації;
- дати фактичної оплати;
- службових нотаток.

Підтримуються такі методи оплати:

- `no_method` - спосіб ще не вибрано;
- `cash` - готівка;
- `card` - картка;
- `transfer` - банківський переказ.

Статуси оплат:

- `draft` - чернетка платежу;
- `scheduled` - запланований банківський переказ;
- `paid` - оплачено;
- `overdue` - строк оплати минув;
- `canceled` - платіж скасовано;
- `refunded` - платіж повернено.

Статуси оплат обмежуються методом оплати:

- для `no_method` доступні `draft`, `overdue`, `canceled`;
- для `cash` і `card` доступні `paid`, `canceled`, `refunded`;
- для `transfer` доступні `scheduled`, `paid`, `overdue`, `canceled`, `refunded`.

Коли платіж переходить у статус `paid`, інші незавершені платежі цього ж полісу зі статусами `draft`, `scheduled` або `overdue` переводяться в `canceled`.

У таблиці полісів відображається не просто останній платіж за id, а актуальний стан оплати за пріоритетом:

1. `paid`
2. `scheduled`
3. `overdue`
4. `draft`
5. `refunded`
6. `canceled`

Це потрібно для того, щоб випадковий пізніший draft або canceled-платіж не перекривав фактичний оплачений стан полісу.

### 5. Страхові випадки

Модуль страхових випадків дозволяє:

- створювати заяву по активному або завершеному полісу;
- перевіряти, що дата події входить у строк дії полісу;
- фіксувати дату події;
- вказувати місце, причину та опис;
- вносити заявлену, резервну та виплачену суму;
- вести нотатки по заяві;
- відстежувати статус кейсу.

Страховий випадок можна створити:

- для активного полісу;
- для завершеного полісу, якщо дата страхового випадку була в межах строку дії полісу.

Для скасованих або draft-полісів створення страхового випадку недоступне.

Статуси страхових випадків:

- `на розгляді`;
- `схвалено`;
- `виплачено`;
- `відхилено`.

Для сум діють перевірки:

- заявлена сума повинна бути більшою за 0;
- заявлена сума не може перевищувати суму покриття полісу;
- резервна сума не може перевищувати суму покриття полісу;
- виплачена сума не може перевищувати резервну суму;
- для статусу `виплачено` потрібна виплачена сума більше 0;
- для статусу `відхилено` виплачена сума повинна дорівнювати 0.

Нотатки до страхових випадків зберігаються як історія роботи по кейсу.  
Нові нотатки можна додавати під час редагування страхового випадку, але вже створені нотатки не редагуються і не видаляються через форму.

Це захищає внутрішню історію комунікації від переписування чужих коментарів.

### 6. Журнал дій користувачів

У системі реалізовано журнал дій користувачів, який дозволяє:

- логувати створення, оновлення та видалення записів;
- бачити історію змін;
- обмежувати видимість логів відповідно до ролі користувача.

> Важливо: журнал дій фіксує саме ручні дії користувачів у CRM: створення, редагування та видалення записів через адміністративну панель.
> Автоматичні службові операції, наприклад щоденний перерахунок статусів оплат і полісів командою `policies:daily-sweep`, не вважаються діями співробітників і не записуються в цей журнал.
> Такий підхід дозволяє відокремити активність користувачів від системної демо-логіки.

### 7. Страхові довідники

Окремий блок довідників:

- категорії страхувань;
- страхові компанії;
- страхові продукти;
- страхові пропозиції.

Для manager довідники доступні лише в режимі перегляду.  
Для admin та supervisor доступні створення, редагування та видалення.

## Дашборд

У системі реалізовано рольовий дашборд.

### Для manager:

- кількість власних клієнтів;
- кількість власних полісів;
- кількість власних страхових випадків;
- прострочені оплати по власних полісах;
- динаміка клієнтів і полісів;
- поліси, строк дії яких скоро завершується;
- нові вхідні заявки;
- нотатки по власних страхових випадках.

### Для supervisor:

- загальні показники по роботі менеджерів;
- динаміка клієнтів і полісів;
- розподіл полісів за статусами;
- аналітика по менеджерах;
- таблиця полісів, строк дії яких скоро завершується;
- внутрішні нотатки по страхових випадках.

### Для admin:

- загальні системні показники;
- контроль проблемних записів;
- журнал останніх дій користувачів;
- заявки на скидання паролів;
- доступ до адміністративних і довідкових розділів системи.

## Бізнес-логіка статусів

### Статус полісу

Статус полісу визначається автоматично за правилами:

- якщо є оплачений платіж, строк дії полісу вже почався і ще не завершився -> `active`;
- якщо є оплачений платіж і строк дії полісу завершено -> `completed`;
- якщо оплати немає і строк оплати минув -> `canceled`;
- в інших випадках -> `draft`.

Скасований поліс вважається фінальним станом.  
Якщо поліс перейшов у `canceled`, він не відновлюється автоматично. Для нового договору потрібно створити новий поліс.

Якщо поліс переходить у статус `active`, а клієнт ще має статус `lead`, система автоматично змінює статус клієнта на `active`.

### Статус оплати

Оплата залежить від методу та стану проведення:

- `draft` - чернетка платежу без вибраного способу оплати;
- `scheduled` - запланований банківський переказ;
- `paid` - оплачено;
- `overdue` - строк оплати минув;
- `canceled` - платіж скасовано;
- `refunded` - платіж повернено.

Статуси оплат додатково обмежуються методом оплати:

- для `no_method` доступні `draft`, `overdue`, `canceled`;
- для `cash` і `card` доступні `paid`, `canceled`, `refunded`;
- для `transfer` доступні `scheduled`, `paid`, `overdue`, `canceled`, `refunded`.

### Статус заявки

Вхідна заявка має такі статуси:

- `new` - нова;
- `in_progress` - опрацьовується;
- `converted` - конвертовано у клієнта;
- `rejected` - відхилено.

Після конвертації заявки в клієнта повторна конвертація блокується.

### Статус клієнта

Клієнт має такі статуси:

- `lead` - потенційний клієнт;
- `active` - активний клієнт;
- `archived` - архівований клієнт.

Новий клієнт створюється зі статусом `lead`.  
Після активації полісу клієнт автоматично переходить у `active`.

## Особливості демо-логіки

Проєкт розроблявся як навчальний / дипломний проєкт без інтеграції з реальною платіжною системою.

Тому для демонстрації життєвого циклу оплат і змін статусів у системі використовується окрема демо-логіка:

- щоденна команда `policies:daily-sweep`;
- автоматичне переведення частини актуальних запланованих банківських переказів у статус `paid` за допомогою параметра `--rate`;
- автоматичне позначення прострочених оплат;
- автоматичне скасування інших незавершених платежів після появи оплаченого платежу по полісу;
- створення службових сповіщень про прострочені платежі;
- централізований перерахунок статусів полісів.

Цей механізм використовується саме для демонстрації бізнес-логіки в умовах відсутності інтеграції з зовнішнім платіжним провайдером.

## Приклад бізнес-процесу

1. Потенційний клієнт залишає заявку на публічному лендингу.
2. Система валідує заявку та створює запис у CRM.
3. Заявка автоматично призначається активному менеджеру з найменшим навантаженням.
4. Менеджер отримує внутрішнє сповіщення про нову заявку.
5. Менеджер опрацьовує заявку.
6. На основі заявки створюється картка клієнта.
7. Менеджер обирає страхову пропозицію та створює поліс.
8. Система автоматично створює початковий draft-платіж і встановлює строк оплати.
9. Менеджер або відповідальний співробітник фіксує оплату.
10. Система автоматично оновлює статус полісу.
11. Якщо поліс стає активним, клієнт зі статусом `lead` автоматично переходить у статус `active`.
12. У разі страхового випадку створюється заява та ведуться нотатки по кейсу.
13. Supervisor контролює роботу менеджерів і загальну бізнес-картину по відділу.
14. Admin контролює технічний стан системи, користувачів, довідники та проблемні записи.

## Встановлення та запуск

### 1. Клонування репозиторію

```
git clone https://github.com/meowam/finance-crm.git
cd finance-crm
```

### 2. Встановлення PHP-залежностей

```
composer install
```

### 3. Встановлення frontend-залежностей

```
npm install
```

### 4. Підготовка середовища

```
cp .env.example .env
php artisan key:generate
```

### 5. Налаштування бази даних

Для локального запуску проєкту та роботи з демо-даними використовується **MySQL**.

У `.env` потрібно вказати:

```
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=finance
DB_USERNAME=root
DB_PASSWORD=
```

Після цього виконати міграції:

```
php artisan migrate
```

> Примітка: SQLite у проєкті використовується для автоматичних тестів через `phpunit.xml`.

### 6. Наповнення тестовими даними

```
php artisan db:seed
```

### 7. Запуск проєкту

Базовий локальний запуск:

```
php artisan serve
```

Публічний лендинг буде доступний за адресою:

```
/
```

Внутрішня CRM-панель буде доступна за адресою:

```
/admin
```

Для frontend-ресурсів можна окремо запустити Vite:

```
npm run dev
```

Або виконати production-збірку:

```
npm run build
```

## Демо-команди

Для демонстрації автоматичної логіки оплат і сповіщень можна вручну запускати команду:

```
php artisan policies:daily-sweep
```

Або з власним коефіцієнтом переходу запланованих переказів у статус `paid`:

```
php artisan policies:daily-sweep --rate=0.35
```

Це демо-механізм для моделювання змін статусів оплат у навчальному проєкті.
