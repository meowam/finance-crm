# Finance CRM

Finance CRM - це внутрішня CRM-система для страхової компанії, призначена для автоматизації роботи з клієнтами, полісами, оплатами та страховими випадками.

Система орієнтована на щоденну внутрішню роботу співробітників компанії: менеджерів, керівника відділу та адміністратора. Основний інтерфейс реалізований у вигляді адміністративної панелі на Filament.

## Основна ідея проєкту

Проєкт моделює роботу внутрішньої страхової CRM, у межах якої:

- менеджери працюють лише зі своїми клієнтами, полісами та пов’язаними страховими випадками;
- supervisor контролює загальну картину по всіх менеджерах;
- admin має повний доступ до системи;
- для клієнтів можна оформлювати поліси, фіксувати оплату, відстежувати строки дії та реєструвати страхові випадки;
- система підтримує як фізичних осіб, так і юридичних клієнтів.

## Основні можливості

- ведення клієнтської бази;
- оформлення та супровід страхових полісів;
- облік оплат за полісами;
- реєстрація страхових випадків;
- ведення внутрішніх нотаток до заявок;
- розмежування доступу за ролями користувачів;
- контроль строків дії полісів;
- базова аналітика та звітність для керівника.

## Ролі користувачів

У системі передбачені такі ролі:

- **admin** - повний доступ до системи;
- **supervisor** - керівник відділу;
- **manager** - менеджер.

### Базова логіка доступу

- менеджер працює лише зі своїми клієнтами;
- менеджер бачить лише свої поліси;
- менеджер бачить лише ті страхові випадки, які належать до його полісів;
- supervisor бачить усіх клієнтів, поліси, оплати та страхові випадки;
- admin має повний контроль над системою.

## Основні сутності

### Клієнти

У системі підтримуються два типи клієнтів:

- **фізична особа**
- **компанія**

Для клієнтів зберігається:

- тип клієнта;
- статус;
- контактні дані;
- джерело звернення;
- закріплений менеджер;
- нотатки.

### Особливості моделі клієнта

Для **фізичної особи** зберігаються звичайні персональні дані клієнта:

- ПІБ;
- email;
- телефон;
- номер документа;
- ІПН;
- дата народження.

Для **компанії** клієнтом є юридична особа, але в системі також зберігаються дані **контактної особи**, з якою працює менеджер:

- назва компанії;
- ПІБ контактної особи;
- email;
- телефон;
- номер документа контактної особи;
- ЄДРПОУ / податковий номер компанії.

У базі даних податковий номер фізичної або юридичної особи зберігається в одному полі `tax_id`, але його підпис в інтерфейсі змінюється залежно від типу клієнта.

## Поліси

Для кожного полісу зберігаються:

- номер полісу;
- клієнт;
- страхова пропозиція;
- менеджер;
- дата початку та завершення дії;
- сума оплати;
- сума покриття;
- строк оплати;
- статус полісу.

### Логіка оплат

Після створення полісу система встановлює строк оплати за фіксованим правилом:

- **дата початку дії полісу + 7 днів**

Це правило використовується як у внутрішній бізнес-логіці, так і в інтерфейсі створення полісу.

## Оплати

Для полісів ведеться облік оплат:

- метод оплати;
- статус оплати;
- строк оплати;
- сума;
- службова інформація про оплату.

## Страхові випадки

Система підтримує реєстрацію страхових випадків:

- номер заявки;
- пов’язаний поліс;
- хто зареєстрував випадок;
- статус;
- сума вимоги;
- резервна сума;
- фактично виплачена сума;
- опис події;
- додаткові дані.

## Нотатки до заявок

До кожного страхового випадку можна додавати внутрішні нотатки для фіксації деталей, уточнень та супроводу кейсу.

## Поточний функціонал

На даний момент у проєкті реалізовано:

- CRUD для основних сутностей;
- створення та ведення клієнтів;
- створення страхових полісів;
- облік оплат;
- створення страхових випадків;
- нотатки до страхових заявок;
- автоматична генерація номера полісу та номера заявки;
- рольове обмеження доступу;
- базова логіка статусів полісів та оплат.

## Подальший розвиток

Проєкт може бути розширений такими можливостями:

- журнал активності користувачів;
- задачі та нагадування;
- звіти по роботі менеджерів;
- сторінка контролю продовження полісів;
- публічна форма вхідної заявки для нового клієнта;
- експорт звітів;
- сторінки аналітики та KPI;
- перевірка якості даних і пошук дублікатів.

## Приклад бізнес-процесу

1. Менеджер створює нового клієнта або отримує вхідне звернення.
2. Для клієнта підбирається страхова пропозиція.
3. На основі пропозиції оформлюється поліс.
4. Система встановлює строк оплати.
5. Менеджер або відповідальний співробітник фіксує оплату.
6. У разі настання страхового випадку створюється заявка.
7. Supervisor переглядає загальну картину, навантаження менеджерів та ключові показники.

## Встановлення та запуск

### 1. Клонування репозиторію

```bash
git clone https://github.com/meowam/finance-crm.git
cd finance-crm
2. Встановлення PHP-залежностей
composer install
3. Встановлення frontend-залежностей
npm install
4. Підготовка середовища
cp .env.example .env
php artisan key:generate
5. Налаштування бази даних

За замовчуванням можна використовувати SQLite:

touch database/database.sqlite
php artisan migrate

За потреби можна налаштувати MySQL або іншу підтримувану СУБД через .env.

6. Наповнення тестовими даними
php artisan db:seed
7. Запуск проєкту
composer run dev

Або окремо:

php artisan serve
npm run dev
Вхід до системи

Адміністративна панель доступна за адресою:

/admin

Для входу потрібен активний користувач із роллю:

admin
supervisor
manager
Структура системи

Основні модулі проєкту:

Clients - робота з клієнтами;
Policies - оформлення та супровід полісів;
Policy Payments - облік оплат;
Claims - страхові випадки;
Claim Notes - внутрішні нотатки до заявок;
Users - користувачі та ролі.
